## Temuan Penting

### Nginx rule folder sensitif belum benar aktif

Beberapa path sensitif masih mengembalikan HTTP 200:

- `/application/`
- `/system/`
- `/storage/`
- `/packages/`
- `/composer.json`
- `/composer.lock`

Beberapa file PHP memang menampilkan `No direct script access`, tetapi file seperti `composer.json` dan `composer.lock` masih dapat dibaca publik. Ini perlu diperbaiki sebelum production.

Penyebab: rule Nginx sebelumnya memakai `location /(application|...)`, yang dianggap prefix literal, bukan regex. Harus diganti menjadi regex `location ~ ^/(...)`.

## File Nginx yang Sudah Disiapkan

Config yang sudah diperbaiki tersedia di:

```txt
/home/efendi/.openclaw/workspace/company-profile-v2.nginx
```

Perubahan penting:

```nginx
location ~ ^/(application|cgi-bin|packages|storage|system|vendor)(/|$) {
    return 403;
}

location ~ ^/composer\.(lock|json)$ {
    return 403;
}
```

## Perintah yang Perlu Dijalankan dengan Sudo

Karena akses sudo tidak tersedia dari sesi Discord, jalankan di terminal server:

```bash
sudo cp /home/efendi/.openclaw/workspace/company-profile-v2.nginx /etc/nginx/sites-available/company-profile-v2
sudo nginx -t
sudo systemctl reload nginx
```

Setelah itu audit ulang path sensitif.
